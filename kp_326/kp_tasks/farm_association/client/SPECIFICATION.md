# Спецификация клиента фермерской ассоциации

## Цель

В корне репозитория создается второй проект `client`: React + TypeScript приложение для работы с сервером из каталога `server`. Клиент предоставляет интерфейс к основным бизнес-процессам фермерской ассоциации: учет хозяйств, участников, участков, продукции, заявок, закупок, поставщиков, договоров и административных справочников.

Приложение должно открывать разный набор экранов и операций после авторизации пользователя. Полные права получает роль `agriculture_admin` - администратор сельского хозяйства. Остальные роли ограничены своей зоной ответственности.

## Архитектура

```text
Browser
  -> React UI
  -> client/src/api/FarmApiClient
  -> Vite proxy /server-api
  -> server HTTP API
  -> PostgreSQL fasc_db / fasc_test
```

В режиме разработки клиент не обращается к серверу напрямую с другого origin. Запросы идут через Vite proxy:

```text
/server-api/auth/login -> http://127.0.0.1:8080/auth/login
/server-api/api/farm  -> http://127.0.0.1:8080/api/farm
```

Боевой URL сервера задается переменными:

```text
FARM_SERVER_URL
VITE_FARM_SERVER_URL
VITE_API_BASE_URL
```

## Текущий контракт сервера

Авторизация:

```http
POST /auth/register
POST /auth/login
```

Тело запроса:

```json
{
  "name": "user",
  "password": "password123"
}
```

Успешный ответ:

```json
{
  "token": "<jwt>",
  "token_type": "Bearer",
  "user": {
    "id": 1,
    "name": "user"
  }
}
```

Основные ресурсы доступны по схеме:

```http
GET    /api/<resource>
POST   /api/<resource>
GET    /api/<resource>/item?id=<id>
PUT    /api/<resource>/item?id=<id>
DELETE /api/<resource>/item?id=<id>
```

Списочный ответ:

```json
{
  "resource": "farm",
  "rows": []
}
```

## Роли

| Роль | Назначение | Функционал |
| --- | --- | --- |
| `agriculture_admin` | Администратор сельского хозяйства | Полный доступ ко всем ресурсам, справочникам, настройкам и отчетам. |
| `association_director` | Руководитель ассоциации | Ассоциации, участники, хозяйства в ассоциации, сотрудники, договоры, заявки и заказы. |
| `farm_owner` | Владелец хозяйства | Профиль хозяйства, владельцы, сотрудники хозяйства, участки, заявки на закупку и продажу. |
| `agronomist` | Агроном | Участки, назначения участков, производство и потребление продукции на участках. |
| `procurement_manager` | Менеджер закупок | Поставщики, цены поставщиков, договоры, заявки на закупку, заказы и позиции заказов. |
| `sales_manager` | Менеджер продаж | Заявки на продажу, продукция, хозяйства и договоры. |
| `farm_worker` | Сотрудник хозяйства | Свои назначения на участки, производственные и расходные записи, базовый просмотр продукции. |

## Правила доступа

Клиент скрывает недоступные разделы и действия. Это улучшает UX, но не является защитой данных. Сервер должен дополнительно проверять роль по JWT и отклонять запрещенные операции.

Текущее ограничение backend: `AuthView` возвращает только `id` и `name`, без роли. Поэтому первая версия клиента хранит выбранную рабочую роль в клиентской сессии. Следующий backend шаг:

- добавить роль пользователя в модель пользователей;
- вернуть `role` в `/auth/login` и `/auth/register`;
- добавить role claim в JWT;
- реализовать server-side RBAC в handlers/controllers;
- добавить endpoint `/auth/me` для восстановления сессии.

## Экраны

Первая версия клиента включает:

- экран авторизации и регистрации;
- выбор рабочей роли до появления роли в backend;
- рабочее пространство с боковой навигацией;
- сводные показатели доступных модулей;
- таблицу выбранного ресурса через реальный `GET /api/<resource>`;
- состояние загрузки, ошибки API и пустые состояния;
- сохранение сессии в `localStorage`.

Следующие экраны:

- карточка записи с `GET /api/<resource>/item`;
- формы создания и обновления;
- подтверждение удаления;
- фильтры, поиск, сортировка и пагинация;
- журнал действий;
- отчеты по закупкам, продажам, участкам и продукции.

## Нефункциональные требования

- TypeScript strict mode.
- Все обращения к API проходят через `FarmApiClient`.
- Идентификаторы `std::uint64_t` на клиенте типизируются как `number | string`, чтобы безопасно пережить будущий переход API на строковую сериализацию больших ID.
- Даты передаются и отображаются как ISO `YYYY-MM-DD`.
- Ошибки сервера вида `{ "error": "..." }` показываются пользователю без потери HTTP-статуса в API-слое.
- UI должен быть плотным рабочим интерфейсом, а не landing page.

## Критерии готовности первой версии

- `npm install` устанавливает зависимости.
- `npm run typecheck` проходит.
- `npm test` проходит.
- `npm run build` собирает production bundle.
- При запущенном сервере `npm run dev` позволяет авторизоваться и открыть списки ресурсов.
- Набор модулей меняется при выборе разных ролей.
